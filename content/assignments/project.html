1 - Introduction

You will be using the Linux FUSE (Filesystem in Userspace) interface to implement a file system as a user-space process. 2 - Background

We provide some light background and helpfule links on FUSE and the file system format you will be constructing to give you a high-level overview. The template source code is documented fairly thoroughly and you are expected to read it as well for additional context.

2.1 - FUSE

FUSE (Filesystem in Userspace) is an interface for userspace programs to export a filesystem to the Linux kernel. The FUSE project consists of two components: the fuse kernel module (maintained in the regular kernel repositories) and the libfuse userspace library (maintained in this repository). libfuse provides the reference implementation for communicating with the FUSE kernel module.

A FUSE file system is typically implemented as a standalone application that links with libfuse. libfuse provides functions to mount the file system, unmount it, read requests from the kernel, and send responses back. libfuse offers two APIs: a “high-level”, synchronous API, and a “low-level” asynchronous API. In both cases, incoming requests from the kernel are passed to the main program using callbacks. When using the high-level API, the callbacks may work with file names and paths instead of inodes, and processing of a request finishes when the callback function returns. When using the low-level API, the callbacks must work with inodes and responses must be sent explicitly using a separate set of API functions.

— libfuse Documentation

When you use an application that invokes a file system syscall, (such as open, read, write, or close) the kernel will redirect these syscalls to the virtual file system (VFS). When those calls access files in the FUSE file system, they will be forwarded to the FUSE kernel module, which will in turn invoke the corresponding callbacks registered in the user-space process (e.g. fsx492). This can be seen in Figure 1. The user-space program must define some subset of callbacks that libfuse expects.

Due to all the redirections, FUSE file systems are much slower than kernel-based file systems; however, they are much easier to implement and debug.

2.2 - FSX492 Description

The FSX492 file system is a simplified deriviative of the UNIX Fast File System. The disk image layout is shown in Figure 2.

Definitions for contents of the superblock, inodes, and directory entries can be found in fsx492.h, and many details are documented in the file system.

There are a few things worth noting here:

  • Inodes are UNIX-style inodes, and many fields are functionally equivalent (see the copy_stat function in fsx492.c). The file system uses the inode to represent both files and directories (type stored in mode field and accessible via the standard mode_t macros). There are 6 direct data block pointers, 1 singly-indirect block pointer, and 1 doubly-indirect block pointer, giving a maximum file size of over 64 MB. This is significantly larger than the test image size of 1MB, but you should still be able to support files on that order.

  • The root directory is always inode 1. Inode 0 is reserved as a null-like value.

  • Directories do not make use of either indirect block pointer, only the 6 direct block pointers. This means that directory sizes are capped at 192 entries.

    • Directory entries (fsx492_dirent) make use of bitfields. Note that bitfield order in the memory layout is not guaranteed to be the same across compilers or architectures, so be careful when doing bitfield unpacking in the real world.

  • Helper functions for the inode and block bitmaps are defined for you. They make use of the fd_set type, used for the select syscall.

3 - Setup

You will need to do develop in your Debian VM as you cannot easily mount the file system within a Docker container. If you know of a way to do so without relying on running a privileged container, please let me know.

  1. Boot your Debian VM to a clean kernel (ending in amd64).

  2. Install the necessary packages for development:

    sudo apt install libfuse3-dev pkg-config fuse3

  3. Confirm that the FUSE kernel module is loaded in Debian: lsmod | grep fuse

  4. Download the file template-project.zip, move it to your shared folder, and unzip this archive with the following command:

    unzip -d "<shared>/<login1>-<login2>-<login3>-project" template-project.zip

    Where <shared> is the path to your shared directory, and each <login#> is a Stevens login (e.g. rtsang1) for each of your group’s members.

  5. Change to your project directory and compile the provided project code. Compilation should produce a fsx492 executable. This is the FUSE client program, which registers the file system callback functions you will be defining.

  6. Run the following commands:

    mkdir ./testfs
    cp data/test.img.bkp data/test.img
    ./fsx492 -d -s --img data/test.img ./testfs

    This will create a new directory in the project at which to mount your file system, create a clean test image from the backup image, and attempt to mount the image at ./testfs. If you are curious what the flags mean, run ./fsx492 --help.

    At this point, the project is still blank and toward the end of the ouptut should be the lines:

    fsx492_init
    failed to read superblock

    This error should be resolved after blkdev.c is implemented.

The following commands will likely be also be helpful at some point:

  • mount | grep "fsx492" # confirm if the file system is currently mounted
  • fusermount -u <path/to/mount/folder> # unmount the filesystem if `fsx492` crashed

4 - Testing and Debugging

After you have implemented blkdev.c, you will will be able to start testing and debugging your code.

We have provided some structure to help with both manual and automated integration testing.

4.1 - Manual

When testing manually, you will simply run the fsx492 program to mount the file system and interact with it via standard command line utilities. When you are done testing the program and wish to unmount it, you can kill the process with Ctrl-C. The FUSE client automatically registers an interrupt handler for the kill signal to gracefully clean up and unmount the file system, so long as it is implemented correctly.

Executingfsx492 without any flags will cause it to run in the background by default, so your logging output (printf and fprintf) will not be visible. To run the executable in the foreground, use the -f or -d flags. When executing in the foreground, you will need to interact with the file system via a separate terminal instance.
Additionally, FUSE clients execute in multi-threaded mode by default, which will make logged output much more difficult to parse. To run in single-threaded mode, use the -s flag. You are not expected to make your implementation thread-safe, so you should pretty much always be using -s.
If your program crashes after mounting, it will leave the mountpoint in a broken state, as the interrupt handler will not run. You must run fusermount -u <mountpoint> to manually unmount the broken filesystem before attempting to mount again.

For convenience, we have included 2 .PHONY targets in the Makefile:

  • make reset will automatically unmount the file system from ./testfs and restore the data/test.img from the backup.
  • make mount will mount the filesystem at ./testfs.

While printf debugging is a perfectly viable strategy for this project, you are also able to run the fsx492 executable under GDB and Valgrind.

To run under GDB:

$ gdb fsx492
GNU gdb (Debian 16.3-1) 16.3
Copyright (C) 2024 Free Software Foundation, Inc.

...

Reading symbols from fsx492...
(gdb) handle SIGINT print nostop pass
SIGINT is used by the debugger.
Are you sure you want to change it? (y or n) y
Signal Stop Print Pass to program Description
SIGINT No Yes Yes Interrupt
(gdb) run -s -d --img data/test.img ./testfs

Attempting to kill the program using Ctrl-C while running it under gdb will not actually kill the program due to how gdb intercepts signals.

To allow our program to interact with signals, we must explicitly tell GDB to (1) forward the signal to the program (“pass”) and (2) prevent GDB from stopping itself while forwarding the signal (“nostop”), as shown above.

If you forget to do this and quit GDB, the cleanup code will not run and you will have to manually unmount the filesystem with fusermount.

To run under Valgrind:

$ valgrind --leak-check=full ./fsx492 -s -d --img data/test.img ./testfs
Valgrind functions by executing the program in an emulated virtual machine, so expect it to run significantly slower.

4.2 - Automated

We have also provided a Python framework for automated tests in test.py, which we will expect you to fill out to exercise all of your implemented functions. You can run test.py --help to see a list of its arguments

The test.py script works by launching 2 threads: one to mount the file system, and one to run the test itself. Tests are defined as global functions whose names are prefixed by test_. They take a single argument mountpoint, which is a string containing the path to the directory where the file system was mounted.

Each test is meant to execute a sequence of file operations (using any standard library modules you like), with failure conditions indicated with assert statements. There are two sample tests provided as an example.

You must run tests one at a time, as they are destructive to the test image. The test script will restore the test image from its backup before executing the test function to ensure repeatability. Therefore, your tests must be written with the assumption that they will be modifying a clean image file.

If you make modifications to the test image that you want to keep (for instance, during manual testing), you will need to save a copy of the image file before you invoke test.py, or the changes will be lost.

The output of test.py will only show logs from the script itself; however, the logs from fsx492 will be saved in a text file named fsx492.log in the same directory.
